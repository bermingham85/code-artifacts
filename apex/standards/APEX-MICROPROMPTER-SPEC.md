## 0. Summary

Apex Microprompter takes one complex idea from Michael and turns it into a structured, traceable, multi-LLM execution plan. Each idea is decomposed into micro-prompts. Each micro-prompt has a unique code, a planned output location, a success criterion, and one LLM assigned to it. Prompts run in waves of up to three in parallel. Every output is stamped, registered, and discoverable via a read-only lookup service called Lisa. The system runs on existing Bermech infrastructure: QNAP NAS for canonical files, Supabase for live coordination, n8n for dispatch, Tailscale for offsite access, Telegram for notifications and lookups.

---

## 1. Design goals

1. One human (Michael) talks to one LLM (the originator) per project. Everything else is automated.
2. No two LLMs ever write the same file at the same time.
3. Every output is traceable back to the prompt that created it, in plain English, via QR code or code lookup.
4. Reuse beats re-creation. Every new project checks the registry before any work starts.
5. Naming is deterministic and shared. No LLM invents names freely.
6. Accessible offsite via Tailscale and Telegram only. No new public-internet layer.
7. Failure is logged, not hidden. Stuck tasks surface to ClaudeClaw automatically.

## 2. Roles and responsibilities

Eight roles, clean boundaries.

| Role | Type | Responsibility | Writes? |
|---|---|---|---|
| Originating LLM | LLM (chat) | Project manager for one project. Decomposes the prompt, approves Reuse Report, accepts final delivery. | Yes — via MCP tools |
| Reuse Agent | LLM | Phase 0. Searches registry, fingerprint matches, external sources. Outputs Reuse Report. | No — read only |
| Location Agent | Script (no LLM) | Decides where each output should live based on rules. | No — advisory |
| Control Agent | LLM | Verifies outputs landed correctly, hashed, stamped, registered. | Yes — updates master |
| Worker LLMs | LLM | Execute micro-prompts assigned to them. | Yes — own mailbox only |
| Naming Authority Service | HTTP endpoint | Returns naming rules and validates proposed names. | No — read only |
| Lisa Lookup | Bot + web | Decodes any code to English. Telegram + Tailscale web. | No — read only |
| n8n Orchestrator | Workflow | Watches Supabase, dispatches work, polls mailboxes, updates master. | Yes — the only writer |
| ClaudeClaw | Telegram bot | Notifies Michael of completions, blocks, cost thresholds. | No — notify only |

Hard rule: **only n8n writes to the master.** All other writes go through n8n via Supabase row updates.

## 5. Codes and naming

### 5.1 Prompt code format

`{prompt_number}.{phase_number}.{micro_number}[.{extension_letter}...]`

Examples:
- `1001` — prompt project, top-level reference
- `1001.4` — phase 4 of prompt 1001
- `1001.4.3` — micro-prompt 3 within phase 4
- `1001.4.3.A` — first emergent micro-prompt spawned by 1001.4.3
- `1001.4.3.A.B` — second emergent spawned by 1001.4.3.A

Rules:
1. Prompt numbers start at 1000 and increment.
2. Phase numbers are 0–8.
3. Micro numbers within a phase start at 1.
4. Letter extensions start at A. After Z roll to AA, AB, etc.
5. Codes are immutable. Never renumber an existing code.
6. Parent of any code is the code minus its rightmost segment.

### 5.2 Phase numbers

| # | Name | Purpose |
|---|---|---|
| 0 | REUSE | Check registry, fingerprint, external sources before any work |
| 1 | RESEARCH | Context gathering, prior-art deep dive |
| 2 | FOUNDATION | Scaffolding, register entries, repo branch |
| 3 | DESIGN | Architecture, schema, interface contracts |
| 4 | BUILD | Implementation |
| 5 | INTEGRATION | Wire into APEX (n8n, Supabase, MCP) |
| 6 | VALIDATION | Tests, governance audit, second opinion |
| 7 | DEPLOYMENT | Push to production hosts, activate |
| 8 | HANDOVER | Docs, register update, skill capture, PR to main |

### 5.3 File naming

`{prompt_code}_{slug}.{ext}`

Slug rules:
- Lowercase only
- Underscores, no hyphens or spaces
- Max 40 characters
- Derived deterministically from the project fingerprint
- No version suffixes in the slug — versioning lives in the archive filename

Examples:
- `1001.3.1_receipt_schema.sql`
- `1001.4.3_extractor_main.py`
- `1001.6.2_governance_audit.md`

### 5.4 Folder naming for projects

`{prompt_number}_{slug}` — e.g. `1001_ocr_receipt_extractor`

### 5.5 Versioning

Outputs are never overwritten. When a prompt is re-run:

1. Old output moves to `archive/{prompt_code}_{slug}_v{N}.{ext}` with the previous version number.
2. New output replaces the live file (no version suffix).
3. Master row updates `output_hash` and increments `output_version`.

Live file = no version suffix. Archived files = `_v1`, `_v2`, etc. The slug-ban rules apply to the slug only, not to the archive version suffix — `v2` is a valid archive suffix but a banned slug.

### 5.6 Other naming domains

The Naming Authority (Section 8) also covers APEX register codes, Supabase tables/columns, n8n workflow names, branch and PR conventions, database row IDs.

### 6.1 Constraints and triggers

- `CHECK (code ~ '^[0-9]{4,}(\.[0-9]+){0,2}(\.[A-Z]+)*$')` — enforces Naming Authority code pattern.
- `BEFORE INSERT OR UPDATE` trigger runs topological sort on `depends_on`. Rejects cycles with explicit error.
- `BEFORE INSERT OR UPDATE` trigger normalises `resource_locks` strings to canonical QNAP paths (e.g. `M:\foo` → `/share/Container/apex-microprompter/foo`).
- `state_version` enforces optimistic concurrency. All updates use `WHERE code = ? AND state_version = ?` pattern.

Project-level rows hold project metadata only. Micro-level rows hold execution data.

## 7. Phases and waves

### 7.1 Wave rules

1. Wave 0 is always Phase 0 (REUSE). No other phase starts until Wave 0 approved.
2. A wave contains micro-prompts whose `depends_on` is fully satisfied.
3. **Maximum wave width: 3 in parallel.** Hard cap. If 4+ are eligible, the lowest-priority waits.
4. Priority order within a wave: `parent_code` set (extensions go first) → lower phase number → lower micro number → time queued.
5. Wave gate: every prompt in the wave must reach `status = Done` AND `review_status = Pass` before the next wave starts.
6. A failure in a wave does not stop the wave. Downstream dependents auto-marked `Blocked`.

### 7.2 Resource lock rule

Two prompts in the same wave with overlapping `resource_locks` are illegal. n8n refuses to dispatch them together — moves one to the next wave.

### 7.3 Review gate

After every `BUILD`, `INTEGRATION`, or `DEPLOYMENT` micro-prompt completes, n8n auto-dispatches a review task to a **different** LLM than the executor. Reviewer reads `prompt_text`, the actual output, and `success_criteria`. Returns Pass / Fail / Rework with notes. Fail or Rework spawns an extension prompt automatically.

## 10. Document control

### 10.1 Location Agent (deterministic script, no LLM)

Given `prompt_code`, `phase_number`, `output_type`, returns:

```
expected_output_location = /share/Container/apex-microprompter/projects/{project_folder}/{phase_folder}/
expected_filename_pattern = {prompt_code}_{TBD_slug}.{ext}
```

Slug filled at execution time.

### 10.2 Control Agent (LLM)

Runs after every micro-prompt completes. Verifies:

1. File exists at `actual_output_location`.
2. Filename matches `naming_authority.filename.pattern`.
3. Output is stamped (Section 11).
4. SHA256 matches `output_hash`.
5. No duplicate in the register against the same fingerprint.

If any check fails → `status = Failed`, `failure_mode = OutputControl`. Extension prompt auto-created to fix it.

## 13. QR codes

### 13.1 Default behaviour

QR generated when `output_type` ∈ {`doc`, `report`, `label`, `policy`} OR `qr_required = true`. Other types: no QR.

### 13.2 Encoding

QR encodes: `http://100.93.145.25:8766/d/{code}`. Resolves over Tailscale on phone or laptop. Offsite without Tailscale → URL fails but the code is always printed beneath the QR, so manual Telegram lookup still works.

### 13.3 Files

- PNG saved to `projects/{project_folder}/qr/{code}.png`
- 300x300 px, error correction level M
- Avery L7160-compatible label PDF generator on demand

### 13.4 Generator

`bin/generate_qr.py` — called by Control Agent after qualifying outputs.

## 14. Embeddings (fuzzy search)

Local Ollama on the GPU workstation. Model: `nomic-embed-text` or similar (final pick during prompt 1005 implementation). Used by:

- Reuse Agent — fingerprint near-match search
- Lisa `/find` command — fuzzy text search

No external embedding API calls. Keeps prompt content off third-party servers and zero per-call cost.

## 16. n8n workflows

Six workflows, all named `Microprompter-{title}`:

1. **Microprompter-Master Watcher** — polls Supabase every 30s for queued tasks with satisfied dependencies. Dispatches up to wave cap 3.
2. **Microprompter-LLM Mailbox Sync** — pushes assigned tasks to per-LLM JSON mailboxes. Pulls completed work back.
3. **Microprompter-Review Dispatcher** — auto-creates review tasks after BUILD/INTEGRATION/DEPLOYMENT completions.
4. **Microprompter-Control Gate** — runs Control Agent on every completed output.
5. **Microprompter-Stuck Notifier** — checks stuck tasks, fires ClaudeClaw alerts.
6. **Microprompter-Daily Rollup** — cost ledger summary, master.xlsx regeneration, archive snapshot.

Workflow JSONs version controlled at `services/n8n_workflows/Microprompter-*.json`.

**n8n version pinned** in docker-compose.yml to a specific image tag (e.g. `n8nio/n8n:1.78.0`). No `:latest`. Quarterly upgrade review.

## 17. Per-LLM mailbox pattern

Each LLM has a JSON file under `data/per-llm/`. **JSON, not xlsx** — avoids SMB lock issues entirely. Only n8n reads and writes these.

Mailbox format:

```json
{
  "llm": "Claude Code",
  "updated": "2026-05-29T14:32:00Z",
  "open_tasks": [
    {
      "code": "1001.4.3",
      "mailbox_token": "uuid-nonce-string",
      "prompt_text": "...",
      "success_criteria": "...",
      "expected_output_location": "/share/Container/apex-microprompter/projects/1001_ocr_receipt_extractor/4_BUILD/",
      "depends_on_outputs": [
        {"code":"1001.3.1","path":"...","summary":"..."}
      ],
      "deadline": "2026-05-29T16:00:00Z"
    }
  ],
  "completed": []
}
```

Communication channels:
- **Out:** n8n delivers tasks via the LLM's native channel (MCP tool call, API, Claude Code subagent) with the task data and `mailbox_token`.
- **In:** LLM returns result via the same channel with the `mailbox_token`. n8n matches the token, updates Supabase + the mailbox JSON.

**Mailbox tokens are single-use.** UUID + nonce, server-side revoked on first successful return. Replay rejected.

For LLMs without MCP/API access, the JSON mailbox is the fallback — Michael opens the file, copies the prompt into the LLM chat manually, pastes result back into the same row. Manual but works.

## 18. Reuse Agent (Phase 0)

Runs before any other phase. Workflow:

1. Generate fingerprint from rewritten prompt.
2. Exact fingerprint match against `microprompter.prompt_tasks` → flag matches.
3. Fuzzy match via Ollama embeddings → top 5 closest past projects with similarity scores.
4. External search → GitHub, PyPI, npm, Hugging Face (Context7 MCP). Filter: >50 stars, commit in last 12 months, permissive license.
5. Write `reuse_report.md` to `projects/{folder}/0_REUSE/`.
6. **Always wait for Michael's approval** before Phase 1 starts. (Auto-fork disabled for first 6 months. Revisit policy after the first 10 projects.)

Decision logged to `lineage_parent` and `lineage_type` on the project row.

## 23. Deferred to v1.2 (not blocking build)

The following will land in v1.2 after the OCR Receipt Agent (1001) has run through the full system at least once. They're improvements, not blockers.

1. **Wave timeout rule** — per-task `timeout_minutes` (default 30 for research, 120 for build). On timeout: `status = Failed/Timeout`, downstream `Blocked`, alert.
2. **Retry budget** — `max_retries = 3`, exponential backoff (5min, 15min, 45min), then `Failed/Exhausted`.
3. **Same-LLM review fallback** — after 15 min with no second LLM available, downgrade to same-LLM review with `review_status_note = same_executor`.
4. **`microprompter.reuse_decisions` table** — searchable history of reuse decisions across projects.
5. **`microprompter.prompt_revisions` table** — preserves prompt-text edit history when a prompt is re-run with edits.
6. **Log rotation policy** — daily files (`audit-2026-05-29.jsonl`), compress after 7 days, archive to cold storage after 90 days, delete after 2 years.
7. **Audit log encryption** — for projects flagged sensitive (RTB tenancy, BPCP, estate matters), encrypt audit entries at rest.
8. **Health check endpoints** — `GET /health` on Lisa and Naming Authority. ClaudeClaw pings every 5 minutes, alerts on two consecutive failures.

These will be added based on real friction from the first project. Building them now without that signal is premature.
